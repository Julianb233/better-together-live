  <name>Task 1: Create subscription-gate.ts with getUserTier and requireTier middleware</name>
  <files>src/lib/subscription-gate.ts</files>
  <action>
    Create `src/lib/subscription-gate.ts` with:

    1. **SubscriptionTier type**: `'free' | 'try-it-out' | 'better-together'`

    2. **getUserTier(supabase, userId): Promise<SubscriptionTier>**
       - Query `subscriptions` table for most recent record where `user_id = userId` and `status IN ('active', 'trialing')`
       - If no active subscription found, return `'free'`
       - Return the `plan_id` value cast to SubscriptionTier, defaulting to `'free'` if plan_id doesn't match a known tier

    3. **requireTier(minTier: SubscriptionTier): MiddlewareHandler**
       - Define tier ordering: `['free', 'try-it-out', 'better-together']`
       - Get userId from Hono context (`c.get('userId')`) -- this is set by the auth middleware from Phase 1/2
       - If no userId, return 401
       - Create Supabase admin client from `c.env`
       - Call `getUserTier(supabase, userId)`
       - If user's tier index is less than required tier index, return 403 JSON:
         ```json
         {
           "error": "Upgrade required",
           "requiredTier": minTier,
           "currentTier": userTier,
           "upgradeUrl": "/paywall"
         }
         ```
       - Otherwise call `next()`

    Import the Supabase client creator from `../lib/supabase` (use `createAdminClient` which already exists).
    Import Hono types: `import type { MiddlewareHandler } from 'hono'`
  </action>
  <verify>
    - `grep "getUserTier" src/lib/subscription-gate.ts` returns a match
    - `grep "requireTier" src/lib/subscription-gate.ts` returns a match
    - `grep "403" src/lib/subscription-gate.ts` confirms 403 response for free users
    - TypeScript compiles without errors
  </verify>
  <done>subscription-gate.ts exports getUserTier, requireTier, and SubscriptionTier type. Free users get 403 with upgrade URL.</done>

  <files>src/index.tsx, src/api/ai-coach.ts, src/api/video.ts, src/api/intimacy.ts</files>
  <action>
    Apply the `requireTier` middleware to premium-only routes. The gating should happen at the route level in `src/index.tsx` where routes are mounted, OR as route-level middleware in the individual API files.

    **Approach: Apply in index.tsx at mount point** (preferred -- centralizes gating logic):

    In `src/index.tsx`, import `requireTier` from `./lib/subscription-gate`.

    For premium routes, wrap the route mounting with middleware. The pattern depends on how routes are mounted. If using `app.route('/api/ai-coach', aiCoachApi)`, you can add middleware before the mount:

    ```typescript
    // Premium-gated routes
    app.use('/api/ai-coach/*', requireTier('try-it-out'))
    app.use('/api/video/*', requireTier('try-it-out'))
    app.use('/api/intimacy/*', requireTier('try-it-out'))
    ```

    Place these `app.use()` calls BEFORE the corresponding `app.route()` calls so the middleware runs first.

    **Free-tier routes (NO gating needed):**
    - `/api/checkins/*` -- basic check-ins
    - `/api/goals/*` -- basic goals
    - `/api/activities/*` -- basic activities
    - `/api/dashboard/*` -- dashboard data
    - `/api/communities/*` -- community features
    - `/api/feed/*`, `/api/posts/*` -- social feed
    - `/api/users/*` -- user profile
    - `/api/relationships/*` -- relationship management
    - `/api/payments/*` -- payment endpoints themselves (must be accessible to upgrade!)

    **Premium routes (gate with requireTier('try-it-out')):**
    - `/api/ai-coach/*` -- AI coaching
    - `/api/video/*` -- video calls
    - `/api/intimacy/*` -- intimacy challenges

    NOTE: The webhook endpoint at `/api/payments/webhook` must NOT be gated (it has no auth). The existing public route whitelist from Phase 1 should already handle this. Just verify the webhook path is in the public whitelist.

    Do NOT modify the internal logic of ai-coach.ts, video.ts, or intimacy.ts -- the middleware handles gating before the route handler runs.
