<task type="auto">
  <name>Task 1: Install dependencies and create Sentry + logging middleware</name>
  <files>package.json, src/middleware/sentry.ts, src/middleware/logging.ts, src/lib/logger.ts</files>
  <action>
    **Install packages:**
    ```bash
    npm install @sentry/node hono-pino pino
    npm install -D pino-pretty
    ```

    **Create `src/middleware/` directory.**

    **src/middleware/sentry.ts:**
    - Initialize Sentry at module scope (NOT inside middleware function) using `process.env.SENTRY_DSN`
    - Guard with `if (process.env.SENTRY_DSN)` so it gracefully skips in dev when DSN is not set
    - Config: `environment` from `process.env.VERCEL_ENV || 'development'`, `tracesSampleRate` 0.1 in production / 1.0 otherwise, `sendDefaultPii: false`
    - Export `sentryMiddleware()` as a Hono MiddlewareHandler that wraps `await next()` in try/catch
    - On catch: call `Sentry.captureException(error, { extra: { method, url, path } })`, then re-throw so Hono's onError still fires
    - Do NOT capture 4xx errors -- only unhandled exceptions (which are 5xx)

    **src/middleware/logging.ts:**
    - Use `pinoLogger` from `hono-pino` with pino instance
    - Config: `level` from `process.env.LOG_LEVEL || 'info'`
    - In non-production (`process.env.NODE_ENV !== 'production'`), use transport `{ target: 'pino-pretty' }`
    - In production, use raw JSON output (no transport)
    - Export `loggingMiddleware()` function

    **src/lib/logger.ts:**
    - Create a standalone pino logger instance for use outside of request context (e.g., startup logging, cron jobs)
    - Same config as middleware (level from env, pino-pretty in dev)
    - Export `logger`
  </action>
  <verify>
    Run `ls src/middleware/sentry.ts src/middleware/logging.ts src/lib/logger.ts` -- all exist.
    Run `npx tsc --noEmit 2>&1 | grep -i "sentry\|logging\|logger" | head -10` -- check for type errors in new files.
    Run `cat package.json | grep -E "sentry|pino|hono-pino"` -- all dependencies present.
  </verify>

  <name>Task 2: Wire middleware into index.tsx and add health check endpoint</name>
  <files>src/index.tsx</files>
  <action>
    **Add imports to index.tsx:**
    ```typescript

    import { loggingMiddleware } from './middleware/logging'
    import { createSupabaseClient } from './db-supabase'
    ```

    **Register middleware in this order** (before existing middleware):
    1. `app.use('*', sentryMiddleware())` -- must be first to catch all errors
    2. `app.use('*', loggingMiddleware())` -- second to log all requests
    3. Then existing middleware: rateLimitMiddleware, cors, clearLegacyCookies, supabaseCookieRelay, auth

    **Add health check endpoint** (before API route mounts, after middleware):
    ```typescript
    app.get('/health', async (c) => {
      const health: Record<string, unknown> = {
        status: 'ok',
        timestamp: new Date().toISOString(),
        version: process.env.VERCEL_GIT_COMMIT_SHA?.slice(0, 8) || 'dev',
        environment: process.env.VERCEL_ENV || 'development',
      }
      try {
        const supabase = createSupabaseClient(c.env as any)
        const { error } = await supabase.from('users').select('id').limit(1)
        health.database = error ? 'error' : 'connected'
        if (error) health.status = 'degraded'
      } catch {
        health.database = 'disconnected'
        health.status = 'degraded'
      }
      return c.json(health, health.status === 'ok' ? 200 : 503)
    })
    ```

    Note: `/api/health` is already in the auth except list from Phase 1. Add `/health` (without /api prefix) to the public route whitelist too, OR place the health route before the auth middleware. Placing it before auth middleware is simpler -- put it right after CORS setup but before the auth `except()` block. Actually, the health endpoint is at `/health` not `/api/health`, so it won't hit the `/api/*` auth middleware anyway. Just place it after all middleware.

    **Add Hono onError handler** for Sentry integration:
    ```typescript
    app.onError((err, c) => {
      // Sentry middleware already captured it, return 500

    })
    ```

    **Update package.json scripts** -- add `"lint"` and `"typecheck"` scripts (needed by Plan 03):

    **Files to migrate (prioritized by traffic/criticality):**
    1. `src/routes/inline-api.ts` -- 5 console.error calls (user CRUD, invite, relationships)
    2. `src/api/supabase-auth.ts` -- 21 calls (auth is critical path)

    4. `src/api/communities.ts` -- 16 calls (highest count in API files)

    For each file:

    - Keep the error message text identical (just restructure the format)
