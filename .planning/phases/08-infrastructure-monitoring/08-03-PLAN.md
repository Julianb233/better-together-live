<task type="auto">
  <name>Task 1: Install ESLint and create flat config</name>
  <files>eslint.config.mjs, package.json</files>
  <action>
    **Install ESLint + TypeScript plugin:**
    ```bash
    npm install -D eslint typescript-eslint
    ```

    **Create `eslint.config.mjs`** using ESLint v9 flat config format:
    ```javascript
    import tseslint from 'typescript-eslint'

    export default tseslint.config(
      {
        ignores: ['dist/', 'node_modules/', '.vercel/', '*.config.*']
      },
      ...tseslint.configs.recommended,
      {
        rules: {
          '@typescript-eslint/no-explicit-any': 'off',       // Too many to fix -- codebase uses `as any` extensively per Phase 3 decisions
          '@typescript-eslint/no-unused-vars': ['error', { argsIgnorePattern: '^_', varsIgnorePattern: '^_' }],
          'no-console': ['warn', { allow: [] }],             // Flag all console.* to track migration to pino
        }
      }
    )
    ```

    IMPORTANT: Set `no-explicit-any` to `off` (not `warn`) because the codebase has hundreds of intentional `as any` casts from Phase 3 database migration decisions. Setting it to warn would create noise without actionable items.

    IMPORTANT: Set `no-console` to `warn` (not `error`) so the build still succeeds. This flags remaining console.* calls for incremental migration to pino without blocking CI.

    **Run ESLint to verify it works:**
    ```bash
    npx eslint src/index.tsx --max-warnings 999
    ```
    Expect warnings (console.* usage) but no config errors. Fix any config issues before proceeding.

    If there are actual lint ERRORS (not warnings) that would block CI, fix the most critical ones. For `no-unused-vars` errors, either fix the unused imports or add `_` prefix to unused params. Do NOT suppress the rule -- fix the violations. If there are more than ~20 errors, set `@typescript-eslint/no-unused-vars` to `warn` temporarily and add a TODO comment.
  </action>
  <verify>
    Run `npx eslint src/index.tsx` -- should complete without config errors (warnings are OK).
    Run `cat eslint.config.mjs` -- verify flat config format with typescript-eslint.
    Run `npm run lint 2>&1 | tail -5` -- should show warning count, not crash.
  </verify>
  <done>ESLint flat config created with TypeScript rules. `npm run lint` runs successfully (warnings for console.* are expected).</done>

<task type="auto">
  <name>Task 2: Overhaul CI/CD pipeline from Cloudflare to Vercel</name>
  <files>.github/workflows/deploy.yml, vercel.json</files>
  <action>
    **Rewrite `.github/workflows/deploy.yml`** to target Vercel instead of Cloudflare Pages.

    Key changes:
    1. Rename workflow: "Deploy to Vercel" (was "Deploy to Cloudflare Pages")
    2. Trigger on push to `main` and `staging`, plus PRs to `main`
    3. **lint-and-test job** (quality gate):
       - Checkout, setup Node 20, npm install
       - `npm run lint -- --max-warnings 0` -- BLOCK on lint errors (warnings about console.* are allowed via rule config being 'warn' not 'error', so --max-warnings 0 only blocks actual errors)
       - Actually, since no-console is 'warn', `--max-warnings 0` would block on console warnings too. Use `npm run lint` WITHOUT --max-warnings flag. The lint script itself will exit non-zero only on errors, not warnings. eslint exits 0 when only warnings exist, exits 1 on errors.
       - `npx tsc --noEmit` -- STRICT, NO fallback `|| echo`. If type errors exist, the job fails. This is intentional -- Pitfall 4 from research.
       - `npm run build` -- build must succeed
    4. **deploy-preview job** (PRs):
       - Needs lint-and-test
       - Uses `vercel pull --yes --environment=preview` + `vercel build` + `vercel deploy --prebuilt`
       - Requires `VERCEL_TOKEN`, `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID` secrets
       - Comment preview URL on PR
    5. **deploy-staging job** (push to staging):
       - Same as preview but uses `--environment=preview` with staging alias
       - Environment: staging
    6. **deploy-production job** (push to main):
       - Uses `vercel pull --yes --environment=production` + `vercel build --prod` + `vercel deploy --prebuilt --prod`
       - Environment: production
    7. Remove ALL Cloudflare-specific steps (wrangler-action, D1 migrations, CLOUDFLARE_API_TOKEN, CLOUDFLARE_ACCOUNT_ID)
    8. Remove notify-failure job creating GitHub issues (over-engineered for solo dev)

    **Vercel CLI commands pattern:**
    ```yaml
    - name: Install Vercel CLI
      run: npm i -g vercel@latest

    - name: Pull Vercel Environment
      run: vercel pull --yes --environment=production --token=${{ secrets.VERCEL_TOKEN }}

    - name: Build Project
      run: vercel build --prod --token=${{ secrets.VERCEL_TOKEN }}

    - name: Deploy to Vercel
      run: vercel deploy --prebuilt --prod --token=${{ secrets.VERCEL_TOKEN }}
    ```

    All Vercel commands need: `VERCEL_ORG_ID` and `VERCEL_PROJECT_ID` as env vars (set via `env:` block), plus `VERCEL_TOKEN` passed via `--token` flag.

    **Check vercel.json** -- if it exists and has correct config, leave it. If it references Cloudflare-specific settings, clean them up. The existing vercel.json should already be correct from the project setup (Hono + @hono/vite-build targets Vercel by default).

    **IMPORTANT:** The typecheck step (`npx tsc --noEmit`) may currently fail due to pre-existing type errors. If `npx tsc --noEmit 2>&1 | wc -l` shows more than 50 lines of errors, add a comment in the workflow acknowledging this and keep the step but add `continue-on-error: true` with a TODO to make it strict once type errors are fixed. This is pragmatic -- we want the gate in place even if it can't be enforced yet.
  </action>
