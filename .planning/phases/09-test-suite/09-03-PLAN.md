    Create `tests/integration/payments.test.ts`.

    **Mock strategy:** Mock `src/lib/supabase` and `src/lib/supabase/server` for DB calls, and mock `src/lib/stripe` for Stripe signature verification. Also mock global `fetch` for Stripe API calls (payments.ts uses raw fetch to Stripe API).

    **Test approach:** Import the payments sub-router, mount on minimal Hono app, test via app.request().

    Read `src/api/payments.ts` fully first to understand:
    - Route paths (POST /checkout, POST /webhook, POST /cancel, GET /status, etc.)
    - Request body shapes (which fields required)
    - How Stripe is called (raw fetch, not SDK)
    - How webhook signature is verified

    **Test suites:**

    1. **POST /checkout (or whatever the create-checkout route is)**
       - Returns 400 when plan/tier is missing
       - Returns 400 for invalid plan tier
       - Returns 200 with checkout URL for valid request (mock fetch to return Stripe session)

    2. **POST /webhook**
       - Returns 400/401 when Stripe-Signature header is missing
       - Returns 400 when signature verification fails (mock verifyStripeSignature to throw)
       - Returns 200 when signature is valid and event is processed

    3. **POST /cancel (subscription cancellation)**
       - Returns 400 when subscription_id is missing
       - Returns 200 on successful cancellation (mock fetch to Stripe)

    **Important:** The payments routes require auth middleware. Check if the auth middleware is applied via the main app or inline. If inline, you may need to mock the auth check. If via main app middleware, the sub-router may not have it -- read the source.

    Use `vi.stubGlobal('fetch', vi.fn())` to mock the global fetch for Stripe API calls, and restore in afterEach.
