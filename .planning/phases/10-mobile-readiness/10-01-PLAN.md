NOTE: The anon key is safe to embed in client code (it's designed to be public, like a Firebase config). Do NOT use the service role key.

<task type="auto">
  <name>Task 2: Rewrite useAuth hook and API client for Supabase Auth</name>
  <files>mobile/src/hooks/useAuth.ts, mobile/src/api/client.ts, mobile/src/types/index.ts</files>
  <action>
1. Update `mobile/src/types/index.ts`:
   - Import `Session` type from `@supabase/supabase-js`
   - Update `AuthState` interface to include `session: Session | null` field
   - Keep existing `user`, `isAuthenticated`, `isLoading` fields for backward compatibility

2. Rewrite `mobile/src/hooks/useAuth.ts`:
   - Import `supabase` from `../lib/supabase`
   - Import `Session`, `User as SupabaseUser` from `@supabase/supabase-js`
   - On mount: call `supabase.auth.getSession()` to restore persisted session, then set state
   - Set up `supabase.auth.onAuthStateChange()` listener to update session/user state on any auth event (sign in, sign out, token refresh)
   - `login(email: string, password: string)` -- calls `supabase.auth.signInWithPassword({email, password})`. Returns `{success: true}` or `{success: false, error: string}`. NOTE: signature changes from `(email, name)` to `(email, password)` -- screens will be updated in Plan 02.
   - `register(email: string, password: string, name: string)` -- calls `supabase.auth.signUp({email, password, options: {data: {name, full_name: name}}})`. Returns `{success, error?, confirmationRequired?}`. The `confirmationRequired` flag is `true` when `data.session` is null (email confirmation enabled in Supabase).
   - `logout()` -- calls `supabase.auth.signOut()`
   - Keep `updateUser` function but update it to use the current session user ID instead of `authState.user.id`
   - Keep `setUser` function for backward compatibility (onboarding flow)
   - Return: `{user, session, isAuthenticated: !!session, isLoading, login, register, logout, updateUser, setUser, refresh}`
   - Map Supabase `User` to the app's `User` type in state: extract `id`, `email`, `user_metadata.name` etc. If the full user profile is needed, fetch from `/api/users/{id}` after auth state change.
   - Clean up return on unmount: `subscription.unsubscribe()`

3. Rewrite `mobile/src/api/client.ts` request interceptor:
   - Import `supabase` from `../lib/supabase`
   - Replace the request interceptor: instead of reading `AsyncStorage` for USER_ID and setting `X-User-ID` header, call `supabase.auth.getSession()` and set `Authorization: Bearer ${session.access_token}` header
   - Remove `X-User-ID` header entirely
   - Keep the `STORAGE_KEYS` constant for `USER_DATA` only (still useful for caching profile data locally)
   - Remove `storeUserId`, `getUserId` methods (Supabase handles token storage)
   - Keep `storeUserData`, `getUserData`, `clearAuth` methods (profile cache, not auth tokens)
   - Remove `createUser` method (registration now goes through Supabase Auth directly, not POST /api/users)
   - Keep all other API endpoint methods unchanged -- they work via the interceptor

IMPORTANT: Do NOT call the backend `/api/auth/supabase/*` routes from mobile. Those are server-side routes for the web app (cookie-based). Mobile uses `supabase.auth.*` directly (token-based).
  </action>

  - `grep "detectSessionInUrl.*false" mobile/src/lib/supabase.ts` confirms critical RN config
  - `grep "createUser" mobile/src/api/client.ts` returns NO results (method removed)
  </verify>
