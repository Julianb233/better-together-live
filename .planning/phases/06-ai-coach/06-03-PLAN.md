<objective>
Add per-user rate limiting and response caching to the AI coach to control costs and prevent abuse.

Purpose: Without rate limiting, a single user could exhaust API budgets. Without caching, identical common questions waste LLM calls. Both are required for production use (AI-04, AI-05).

Output: Rate limiting middleware that returns 429 when exceeded, and a response cache that short-circuits LLM calls for repeated questions.
</objective>

       - Use in-memory Map store: `Map<string, { count: number; resetAt: number }>`

       - Otherwise call next()
       - Import HTTPException from 'hono/http-exception', createMiddleware from 'hono/factory'

       - In-memory Map store: `Map<string, { response: string; tier: string; model: string; expiresAt: number }>`

       - Export `cacheResponse(message: string, response: string, tier: string, model: string, ttlMs = 24 * 60 * 60 * 1000): void` -- store in cache, evict oldest entry if cache size > 500
       - Cache is intentionally in-memory (acceptable to lose on cold start -- caching is optimization, not critical)

    Update `src/api/ai-coach.ts`:

    1. Import `aiCoachRateLimit` from `../lib/ai/rate-limiter`
    2. Import `getCachedResponse`, `cacheResponse` from `../lib/ai/response-cache`

    3. Apply rate limiting middleware AFTER requireAuth but BEFORE route handlers:
       ```typescript
       aiCoachApi.use('/*', requireAuth())
       aiCoachApi.use('/ask', aiCoachRateLimit(20, 60 * 60 * 1000))  // 20 per hour
       ```

    4. In POST `/ask` handler, add cache check BEFORE the LLM call:
       ```typescript
       // Check cache first
       const cached = getCachedResponse(message);
       if (cached) {
         // Still save the user message to history (for completeness)
         await saveMessages(supabase, [
           { user_id: userId, relationship_id, role: 'user', content: message, model_used: null }
         ]);
         return c.json({
           response: cached.response,
           tier: cached.tier,
           model: cached.model,
           cached: true,
           timestamp: new Date().toISOString()
         });
       }
       ```

    5. After getting a successful LLM response, cache it:
       ```typescript
       cacheResponse(message, text, tier, model);
       ```

    6. Add `cached: false` to the non-cached response JSON so the client can distinguish.

    7. The rate limiter middleware runs before the handler, so a rate-limited request never reaches the cache or LLM. This is correct -- rate limits should apply even to cached responses to prevent abuse of the endpoint itself.

    IMPORTANT: The rate limiter must run AFTER requireAuth because it needs userId from context. Order matters.
